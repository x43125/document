# Redis在分布式中的应用

> Redis 的特点是，独立于微服务之外的第三方服务，很多都是利用这个特点来使用的

## 分布式ID

全局 ID 生成器

通过利用第三方这个特点可以实现全局唯一的 ID 发号器，所有需要 ID 的微服务，可以来 Redis 里获取最新的 Id，由于 Redis 天然单线程，也避免了 Id 的重复

## 分布式锁

同样利用独立第三方及单线程的特点，实现的分布式锁，所有需要分布式锁的微服务，挨个来 Redis 领锁，由于单线程，所以必然会有先后，先拿到某个代表所得数据的，则表示拿到锁了

1. 单机 Redis 锁 + 看门狗 + 续命
   1. 原子性：取锁的时候，需要判断当前锁是否被人拿了，需要设置过期时间，这些操作需要放到同一个命令里原子执行，可以使用：set lock value nx ex 30
   2. 释放：锁的释放要放在 try-catch-finally的 finally 里
   3. 过期时间：需要设置过期时间，避免取得锁之后，服务崩溃了，则锁将永远被持有
   4. 唯一标识 + 续期：场景：锁过期时间=30 秒，但业务需要执行 60 秒，锁提前释放了，并且有可能会被后来的请求取得锁，而这时前一个业务执行完，删除所的时候，会把后来的业务的锁给删掉了
      1. 续期（看门狗）：起一个定时器，定时给锁续命：只要客户端活着，没有主动释放，则会一直续命
      2. 唯一标识：每一个业务取得锁的时候都要设置也给唯一标识，删除释放锁的时候比较一下，防止释放了别的业务的锁
   5. 可重入锁：可以通过对Key值比较来判断是否是自己的锁，记录重入次数
   6. 公平锁：可以使用队列实现公平锁
2. 红锁 Redlock：部署多个独立的 Redis 节点，向所有节点发送加锁请求，如果有超过一半的节点加锁成功，且总耗时<锁过期时间，则加锁成功
