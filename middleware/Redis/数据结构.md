## String

### 1 overview

字符串类型的值可以是字符串、数字、二进制，最大不超过512MB

整数用int，短字符串用embstr，长字符串用raw

### 3 内部编码

redis内部每种存储类型都基本上有多种不同的内部编码来适应不同的情况，如更快存取和更节省内存

我们使用命令 `object encoding key` 可以来检查key的内部编码。

#### 3.1 int

> 8个字节的长整型

```sh
127.0.0.1> set key 8653
OK
127.0.0.1> object encoding key
"int"
```

#### 3.2 embstr

> 小于等于39个字节的字符串

```sh
127.0.0.1> set key "hello, world"
OK
127.0.0.1>  object encoding key
"embstr"
```

#### 3.3 raw

> 大于39个字节的字符串

```sh
127.0.0.1> set key "abcdefghijklmnopqrstuvwxyz1234567890!@#$%^&*()_+-={}[];<>,./?"
OK
127.0.0.1> object encoding key
"raw"
```

**redis会根据值的类型和长度来自行决定使用哪种内部编码**

### 4 使用建议

redis 没有明确对键的声明方式，所以如果在开发中不注意规范将会导致在使用的时候非常的混乱，因此建议一定要有统一的规范，如果公司有规范就按照规范来，否则可以使用如下规则或建立自己的规范：

> `业务名:对象名:id:[属性]` 的组合作为键
>
> e.g.: `vs:user:1`



### 5 使用场景

1. 缓存功能：redis最典型也应该是使用最多的场景
2. 计数器：常见如视频播放数，博客阅览数自增
3. 分布式锁
4. 全局唯一 ID
5. 验证码
6. 共享session：分布式web服务将用户的session信息放到redis中进行集中管理
7. 限速： 如限制用户一分钟内收取验证码的次数、令牌桶

## Hash

> key: field: value

内部编码：ziplist、hashtable

字段少且小用ziplist，否则hashtable

#### 使用场景

- 信息：用户信息、商品信息等
- 购物车
- 用户多维行为计数

## List

ziplist、linkedlist、quicklist

3.2 版本后统一使用quicklist（ziplist+linkedlist结合体）

ziplist 是一种内存紧凑的压缩型存储结构

quicklist 是一种双向链表，方便变更，但占用内存

quicklist 是结合了这两种链表的优势，整体是quicklist，每个节点是ziplist

#### 使用场景

- 队列
- 按时间轴的场景：
  - 用户近 10 条发帖
  - 最近的 10 条评论
  - 最新 10 条消息

## Set

intset、hashtable

全整数且数量少用intset，否则hashtable

#### 使用场景

- 去重：抽奖每场每人只能中一个

- 打标签：不要重复打标签


## ZSet

ziplist、skiplist

元素少用ziplist，否则skiplist

### skiplist

- 实现简单

- 查询效率很高
- 天然支持范围查询

核心：随机层数生成机制

每次插入数据的时候：每次随机当前插入的节点的层数；有 25% 的几率向上增加一层，共 32 层

时间复杂度

查询：O(logn)

插入：O(logn)

删除：O(logn)

#### 使用场景

- 排行榜
- 热搜
- 延迟队列

**ZSet = 去重 + 排序 + O(log N) 增删改查，最适合"分数动态变化、需要快速取 Top N"的场景，排行榜是其最经典、最不可替代的应用**



## 不常用结构

### Bitmap - 位图

利用Bit位与二进制的关系，以非常节约内存的形式保存一些布尔类型的数据

如：用户是否签到、用户在线状态、是否点赞等

### HyperLogLog - 误差换取极致空间

比如：想统计一个网站的 独立访客（UV）的时候，如果数据量是亿级别的，那使用 set 来去重保存内存占用量惊人

而如果用hll，则可以将这个内存占用缩小到 12kb （固定占用 12kb）

代价是：数据会有误差，标准误差 0.81%

##### 使用场景

可以接受误差的，海量数据去重计数场景

- 网站独立访客
- 注册IP 数统计

### GEO - 地理空间

能够快速完成 “查找附件” 的操作

### Stream - 消息队列

此前常用list来做消息队列，但他不支持消息确认，消费者挂了消息就丢失了；也不支持广播
