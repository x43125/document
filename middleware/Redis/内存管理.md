# 内存管理



## 过期键删除策略

- 定时删除（Redis未采用）：键过期时间到的时候同步删除；省内存；占cpu；如果遇到大批量一次性删除时会占用cpu
- 惰性删除：过期后先不删除，等待再次获取该键的时候再删除；占内存；省cpu；如日志等不适合
  - expireIfNeeded: 所有读写数据库的Redis命令再执行之前都会调用 expireIfNeeded 函数检查输入键
    - 如果已过期则删除
    - 否则不做动作
    - 像一个过滤器
    - 每个命令的实现函数都必须能同时处理键存在和键不存在两种情况
- 定期删除：每隔一段时间检查一次，将过期键删除；需要平衡间隔时间和删除时常；否则会退化成定时或惰性删除。
  - activeExpireCycle：规定时间内，分多次遍历服务器中各个数据库，从数据库的expires字典中随机检查一部分键的过期时间，并删除其中过期键
    - 函数每次运行时，从一定数量的数据库中随机取出一定数量的键进行检查，删除其中过期的
    - `current_db`会记录当前函数检查进度，等待下次时接着进行
    - 当所有数据库都被检查一遍后，`current_db`变量会重置为0，开始新一轮

> 已过期的键不会被保存到RDB文件中

一般都是二者结合：惰性删除 + 定期删除

## 内存淘汰策略

过期策略只能处理已经过期的数据，如果确实还有很多的键未过期占满了内存，新键就无法继续创建了，这个时候就会触发淘汰策略，先清理已过期的键，如果还不行，再执行淘汰策略

- noeviction 不淘汰，写入报错
- allkeys-lru 所有键中选择 最近最少使用的数据
- allkeys-lfu 所有键中选择 最不经常使用的数据
- allkeys-random 所有键中随机选择删除
- volatile-lru 所有设置了过期时间的键中选择最近最少使用的数据
- volatile-lfu 所有设置了过期时间的键中选择最不经常使用的数据
- volatile-random 所有设置了过期时间的键中随机删除
- volatile-ttl 所有设置了过期时间的键中选择最快将被淘汰的数据

## 内存分配

内存分配器：Jemalloc

- 线程本地缓存，减少锁竞争
- 内存碎片率低

内存占用情况：

```
Redis 总内存（used_memory）：
┌─────────────────────────────────────────────────┐
│                                                 │
│  实际数据(key + value)        约 60%~80%          │
│  元数据(对象头、过期时间等)      约 10%~20%         │
│  缓冲区(客户端、复制、AOF)      约 5%~15%           │
│  内存碎片                     约 5%~15%           │
│  其他(Lua、模块、内部数据结构)   约 2%~5%            │
│                                                 │
└─────────────────────────────────────────────────┘
```
