### 复杂系统

#### 什么叫复杂系统，怎么定义的？

答：架构、业务、数据、性能、运维、工程

逐字稿：我觉得可以从几个方面来考虑，比如说：架构复杂度、业务复杂度、数据复杂度、性能要求复杂度、运维复杂度和工程复杂度。分别来说的话就是：

1. 架构复杂度指的是需要复杂的架构才能支撑的系统，如包含多个分层，服务节点数量多，跨机房/地域部署，混合云，包含多种通信模式，同步比如http\rpc直接调用，异步包括mq等，是否涉及多种中间件等。
2. 业务复杂度指的是业务线比较多，或单条业务线比较复杂涉及多个方面，如包含几百个微服务这种就是比较直观的一个业务复杂度的体现。服务之间关联复杂，一个服务可能依赖几个甚至几十个其他服务，各个服务之间的关联错综复杂，形成一种依赖网，其他还可以包括如状态机复杂，比如一个审批流的状态可能有十几种。
3. 数据复杂度这个理解起来相对要容易一些，比如数据量的大小，数据的生成速度，对数据的存储要求，实时处理，批处理数据量，一致性要求。
4. 性能复杂度：吞吐量，是否有QPS超过10万的接口或是有没有TPS超过百万的事务，延迟要求比如要求低于50ms
5. 运维复杂度：比如是否需要金丝雀模式部署，是否需要容器/虚拟机混合部署，从指标维度又包括服务的暴露的指标数量，精度等，日志规模及保存时间等要求
6. 工程难度：代码规模比如超百万行代码，协作比如需要多个团队协作实现，跨时区完成等等又或者需要多个大版本迭代。

总之，在我的工作的过程中有涉及到包含其中部分的复杂型项目，如怪兽的项目就是需要多个团队协作，多的时候几十个微服务协作，并且需要k8s+docker等部署的分布式集群架构，以及拥有亿级数据量及大量日志规模等。

  - 调整新生代和老年代的比例； -XX:NewRatio参数来设置新生代的比例，优化晋升频率
  - 调整新生代的分区的比例：-XX:SurvivorRatio参数和-XX:MaxTenuringThreshold参数来调整Eden区、Survivor区的比例，优化minor gc的频率

###### eureka

eureka是一个服务注册、发现组件，他为微服务之间的调用提供一个基本的媒介，本质上是一个分布式的map，用来存储各个服务的基本信息。

主要包括以下几个部分：

- 服务注册：如果已经有了则更新注册时间，否则注册（其实就是存到map里）
  - map格式：ConcurrentHashMap<serviceName, Map<instanceCode, Lease<InstanceInfo\>>>
- 服务下线：
  - 定时器判断：当前时间是否大于过期时间：当前系统时间>lastUpdateTimestamp + 30s
  - 一次下线太多会限制只剔除15%（默认可配置）
  - 随机剔除
- 心跳机制
  - 更改最后操作时间
- 集群同步
  - 如果是一个client node注册到server node上，这个server会把client转发到其他的server上实现集群间的同步
  - 具体又一个switch case来处理各种不同的事件
- 自我保护机制
  - 如果短时间内，大量的心跳链接过期，也就是大量宕机，就会保护，大量的阈值是 85%
- 3层缓存
  - 只读缓存：ConcurrentHashMap
  - 读写缓存：guava
  - 真是数据：ConcurrentHashMap



总结：

Eureka 是一个比较优秀的服务注册中心， 实现了 AP 主要是保证可用性和分区容错性。

1. Eureka Server 的所有的节点配置信息是存储在内存中的，查询服务服务注册信息使用了多层缓存。
   - 多层缓存：首先会进行一级缓存 `**readOnlyCacheMap**`读取，然后读取二级缓存`**readWriteCacheMap**`,  最后读取真实数据
   - 缓存过期，在接收到 register , renew  cancel 请求后会失效二级缓存；服务剔除会删除二级缓存；二级缓存本身过期。
   - 缓存更新，一级缓存中查询不到的时候，会从二级缓存中查询， 如果二级缓存中存在，一级缓存中不存在会同步到一级缓存中。定时任务 3分钟也会主动同步一次一级缓存。
2. Eureka Server 集群环境中，注册的服务只会向一台服务发起注册，然后当前服务端节点会遍历其它节点进行注册信息同步。
3. Eureka Client 默认会 30 秒中发送一次心跳来进行续约，告知 Eureka Server  客户端依然存活没有问题，如果 Eureka Server 90 秒钟没有收到客户端的续约，它会将实例从注册表中删除。
4. Eureka Server自我保护机制，当大量的服务过期的时候，存活服务低于 85% 的时候，就会启动自我保护机制，每次只会下线 15%的服务。

然后我来讲下hystrix

###### hystrix

hystrix的主要几个功能有：

1. 服务降级：在高并发场景下避免用户一直等待，让用户体验更好（返回一个友好的提示，直接给客户端，不会去处理请求，调用fallback方法），比如秒杀：“当前请求人数过多，请稍后重试”；
2. 服务熔断：为了保护服务，在高并发下，如果请求达到了一定极限或流量超过了设置的阈值，直接拒绝访问，保护，使用服务降级的方式返回一个友好地提示，所以服务熔断和降级经常一起搭配着使用；
3. 服务隔离机制：默认情况下只有一个线程池维护所有的服务接口，如果大量的请求访问同一个接口，会导致线程池全被占用，其他的接口无法访问。所以需要隔离开来，相当于每个接口都有一个线程池
   1. 实现方式：线程池：每个接口都有独立的线程池互不影响
   2. 信号量：计数法，为当前接口的线程运行数记录一个数，超过则拒绝，不超过则通行。



一些基本常识：

1. 熔断器支持自动运行和手动调用，来停止当前依赖一段时间，默认10秒。自动的逻辑是当错误率超过阈值则启用，默认是50%
2. 依赖调用超时时间设置：设置比99.5%的平均时间略高

工作原理：

1. 熔断设计
   1. 请求判断算法：使用无锁循环队列计数，每个熔断器默认维护10个bucket，1秒1个，每个bucket记录请求的成功、失败、超时、拒绝等状态，**默认错误超过50%且10秒内超过20个请求的**中断拦截。
   2. 熔断恢复：对于被熔断的请求，每隔5秒允许部分请求通过（打开一点阀门），如果请求都是健康的，则恢复请求（全打开阀门），如果依然有失败的，则继续拦截（继续关闭阀门），等待下一个5秒。
   3. 熔断报警：熔断的请求需要打日志，当异常请求触达一定的阈值则报警。
2. 隔离
   1. 线程隔离：使用一个线程池来存储当前的请求，线程池对请求作处理，设置任务返回处理超时时间，超出线程的请求推入队列中，并设置等待时间
   2. 信号量：使用一个原子计数器来记录当前有多少个线程在运行，请求到的时候先判断计数器数值，如果超过最大限制则丢弃，若未超过则将该请求的计数器+1，请求返回的时候-1，好处是无需额外的线程池处理，缺点是无法应对大量请求，超出必须立刻拒绝
3. 超时机制
   1. 等待超时：在任务队列设置任务入队时间，并判断队首的任务入队时间是否大于超时时间，超过则丢弃
   2. 运行超时：直接可使用线程池提供的get方法
4. 命令模式：



工作逻辑：

两个窗口

一个跳闸

一个试探



- 统计窗口（10秒）：10秒内如果达到了最小请求数，且错误请求超过了阈值（50%）则跳闸，否则都不算
- 试探窗口（5秒）：当跳闸后，每5秒放一个请求过去，如果成功则开闸恢复，如果依然失败则继续关闸重复试探

###### ribbon

ribbon的作用是负载均衡

负载均衡策略：

- RoundRobinRule：轮询：递增并根据实例数量取模，取出来的即是目标实例，并将该数值赋值给原数据
- RandomRule: 随机：使用所有实例数作为种子，随机出下一个值，使用即可
- AvailabilityFilteringRule: 可用性过滤规则：底层使用RoundRobinRule，然后预测节点是否可用，如不可用就继续选择下一个，他会过滤掉因为超访问而处于断路状态及并发访问超出阈值的节点
- BestAvailableRule:最佳：类似AvailabilityFilteringRule会先过滤掉失效节点，再选择一个并发量最小的
- WeightResponseTimeRule:根据响应时间来设置权重，响应时间越快的，权重越大
- RetryRule:重试：底层是RoundRobinRule，如果获取的不可用则在指定时间内会重试
- ZoneAvoidanceRule：
- 自定义：实现IRule接口，复写choose方法，通过@Configuration\@Bean将自定义rule注入到Spring中，替换默认的RoundRobin，还可以通过@RibbonClient来固定只对某个接口生效。

#### 我看你还说你对RocketMQ也很熟悉，讲一讲你们一半都在哪些地方用的RocketMQ

首先mq的主要使用场景有以下这几种

- 异步处理
- 削峰/限流
- 降低系统耦合性/订阅

然后在我们的系统里，mq的主要使用场景有：

- 削峰/限流
  - 削峰：更新ES数据：业务通过向指定topic中传入一条消息来告知消费端要更新哪个es index，更新方式。
  - 限流：通过RocketMQ可以实现限流场景，可以结合令牌桶、或是手动控制每个消费的时长来限流
  - 定时器：结合xxl-job实现分布式定时器，该定时器每当到达定时的时候会像订阅的微服务的所有实例中的一个发送一条mq，来触发定时器执行任务。
- 降低系统耦合性/订阅：
  - 比如人员系统中，当每次hr新增了一个人员的时候，会触发一个新建人员的mq，各业务系统则根据自己的系统需求去订阅这个mq，来实现对应的功能。
  - 又或者比如地址变更，有些地区有的时候会有行政层面的变更，比如一些县可能会变成市，一些区可能会合到一起，这个时候也会触发一个mq由各个业务系统自己去订阅改变来同步地区的变更。
  - 比如审批流系统，审批流程往往涉及到的状态丰富，节点自由，每到一个节点，状态每次变更比如审批通过，审批驳回等等的时候，业务需要相应的做出逻辑，这个时候也是通过监听mq来处理的。
- 异步处理：比如pdf转图片、大文件上传，通过先发起一个异步的上传过程，然后再去监听处理结果。



##### 那你们为什么选择RocketMQ呢？

我们也并不是只有RocketMQ，像我们的大数据处理、日志收集、清洗等就是使用的Kafka，在这方面Kafka更加擅长一些。

目前市场上主要是ActiveMQ\RabbitMQ\RocketMQ\Kafka四种

- ActiveMQ比较简单，性能最差，为万级
- RabbitMQ也是吞吐量万级的MQ，并且RabbitMQ是使用Earlang开发的，在国内这种语言用的比较少，但他的并发能力很强，延时很低，到微秒级
- RocketMQ的吞吐量很高达到十万甚至百万级，然后它是阿里开源的项目，稳定性经受了阿里多年的双十一考验，并且它是使用Java开发的，比较方便阅读和定制源码。
- Kafka只提供了核心功能，但提供超高的吞吐量、ms级别的延迟，极高的可用性和可靠性，而且方便扩展，在大数据处理、日志收集上非常适合



消息队列的功能的举例：异步、解耦、削峰

买票之后给用户发送短信、邮件、微信等通知

首先买票之后不用同步的调用短信、邮件、微信等发送操作，改为异步调用节约时间。

其次当又有新的发送通知方式进来的时候，不用修改代码，而是改成订阅形式，每次只写自己的核心代码，订阅相应的mq即可

最后是，如果像春节期间抢票潮，如果大量的流量打到短信系统，短信系统可能会撑不住，这时候就需要mq来削峰了，大量的发送短信任务进入到mq后，短信系统尽己所能的去消费，而不需要一下接受大量的请求



##### 那mq有什么坏处呢？

没有哪一项技术是“银弹”

引入了mq后，

- 首先系统无疑是变得更加复杂了的
- 编程也变得绕了一层
- 需要考虑mq宕机的情况，于是需要考虑高可用
- 消费失败的问题
- 重复生产消息的问题
- 重复消费的问题
- 消费者太慢了，或者宕机了，导致积压怎么办



##### 怎么解决这些问题

1. 如何避免重复消费
   1. 幂等：将消费者设计成幂等的，比如可以通过引入redis，redis的key,value天生幂等；亦或者是通过为数据库设置唯一索引来控制插入的时候可以自动控制；最重要的还是根据业务来控制，每条业务设置一个唯一ID，每次消费开始的时候根据唯一id+状态的方式来控制是否已被消费，最好能加上分布是锁，避免并发。
   2. 合理设置重试次数，频率
2. 消息积压怎么办
   1. 判断积压是否正常，如果只是合理积压可以不管，等待慢慢消费
   2. 增加消费者和队列
   3. 增加消费者线程数
   4. 检查消费者是否有硬件方面如：CPU、内存、网络等异常
   5. 检查消费者是否有异常问题，比如报错、死锁、死循环等异常问题，有的话需要紧急处理
