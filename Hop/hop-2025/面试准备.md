# Redis

## 一、常见问题

### 1.1 缓存穿透

- 问题：突然大量的请求缓存不存在的数据，导致大量请求直接打到数据库
- 原因：恶意攻击，业务逻辑问题：失败无限重试、或其他频繁访问场景
- 解决方案：
  - 对所有请求的数据都设置缓存，如：get(null)，就把null也缓存到redis中，下次直接读取null就返回
  - 布隆过滤器，通过使用布隆过滤器来先预判key是否存在，key不存在的就直接返回，存在的再去查
  - 接口层增加参数校验如：ID范围、格式

### 1.2 缓存击穿

- 问题：某个热点key突然失效，导致大量的请求直接打到数据库
- 原因：key过期时，并发量大
- 解决方案：
  - 热点key永不过期，通过逻辑主动更新
  - 使用互斥锁，保证同一时间只会有一个线程重建任务
  - 采用多级缓存 ？？？原理

### 1.3 缓存雪崩

- 问题：大量key同时失效或redis宕机，导致大量请求直接打到数据库上
- 原因：缓存过期时间设置的过于统一；redis不是集群模式或集群故障
- 解决方案：
  - 缓存过期时间设置的随机一些，比如可以在每个key的过期时间后加一个随机时间
  - redis集群一定要采用集群部署模式：哨兵模式、Cluster模式，保障高可用
  - 降级策略：采用多级缓存，多加一层缓冲
  - 限流：对于访问量大的请求，对数据库的请求限流

### 1.4 数据一致性

- 问题描述：缓存与数据库的数据不一致（更新数据后，缓存未更新）
- 原因：并发写入、延迟删除或操作失败
- 解决方案：
  - 先更新数据库，再删除缓存，设置过期时间
    - 延时双删：更新数据库后删除redis，延时后再删除redis
      - 为什么不直接延时后再删除，要先删除一次？：为了保证数据的尽量一致，如果不先删除，那么在删除之前读取的就都是旧数据了。
      - 为什么要延时在删除一次？避免上面的一次删除在并发场景下产生的旧数据被更新到缓存中的场景

  - 异步更新：、mq
    - binlog订阅：业务只负责读和更新数据库，单独的同步redis逻辑通过订阅binlog来更新redis，可以实现最终一致性
    - mq：类似binlog订阅，每次更新数据库的时候发送更新操作，消费者消费更新操作来同步redis

  - 强一致：
    - 写操作强制走数据库，禁用缓存短时间内的写入
    - 缓存的写入加版本号机制，或乐观锁


### 1.5 大key问题

- 问题：单个key的value过大如MB级String，或百万元素的Hash等
  - 阻塞Redis（单线程模型）
  - 网络传输延迟高
  - 内存不均
- 解决方案：
  - 拆分大key
  - 压缩value
  - 避免使用keys * 类阻塞命令
  - 业务设置定时任务定时检测是否有大key，扫描到则通知整改

### 1.6 热Key问题

- 问题：某个key被高频访问，导致redis单节点负载过高
- 解决方案：
  - 本地缓存热key
  - 对key进行切片，再细分
  - 使用redis集群分散请求

### 1.7 内存溢出与淘汰策略

- 问题：Redis内存占满，触法淘汰策略或写入失败
- 原因：未设置淘汰策略或策略不当
- 解决方案：
  - 配置合理的maxmemory-policy(volatile-lru)
  - 监控内存使用，及时清理无用key
  - 限制客户端使用内存大小
  - 规定所有key都设置过期时间

### 1.8 持久化阻塞

- 问题：RDB快照或AOF重写，占用大量资源导致服务延迟
- 解决方案：
  - 主从分离，在从上持久化
  - 合理配置AOF重写阈值
  - 使用混合持久化机制

### 1.9 网络与连接问题

- 链接池耗尽
- 超时设置不合理
- Redis集群节点故障转移时的链接中断
- 解决：
  - 合理配置链接池参数
  - 重试机制+断路器
  - 客户端支持集群自动切换

### 1.10 并发竞争

- 多个客户端同时修改一个key
- 解决：
  - 使用乐观锁
  - 分布式锁
